想從終端機直接看 GitHub Actions（自動執行測試、建置或部署工作的服務），現在 gh browse 多了一個 --actions 旗標（flag，附在指令後改變行為的選項）。它讓瀏覽指令能指定開啟 Actions 頁面，不必先進專案首頁再手動切換分頁。公告只說新增這個旗標，沒有承諾改變工作流程的內容或結果。就像電梯多了一顆直達地下停車場的按鈕：目的地沒變，只是少按幾次樓層鍵。

Browse straight to Actions

When you want to inspect GitHub Actions—the service that runs automated tests, builds, or deployments—from the terminal, gh browse now has an --actions flag. A flag is an option added to a command to change what it opens, and this one sends the browse command to the Actions page instead of making you start on a repository page and switch tabs. The notes only add this shortcut; they do not claim to change what workflows do or produce. Like adding a direct elevator button for the parking garage: same destination, fewer buttons.

Use a Git remote name to choose the default repository

When setting the default repository with gh repo set-default, you can now name a Git remote, such as origin. That helps people who already gave one codebase several remote destinations choose the default by the label they recognize. The notes do not say the command will create or rewrite any remotes for you. It is like choosing an existing “work” or “personal” label in an address book instead of copying the full address again.

Unexpected project data no longer makes the tool stop abruptly

When working with GitHub project data, null project items—places where an item exists but has no readable value—will no longer trigger a panic, an abrupt program stop when it cannot handle an error. This is a repair to an existing path, not a new project-management feature. The same release also corrected the Debian CLI package link in the installation guide, sending readers to the right place. Like a clerk skipping a blank box on a form and continuing, while the sign at the entrance is pointed the right way.

Git remote 이름으로 기본 저장소를 고를 수 있다.

gh repo set-default로 기본 저장소를 정할 때, origin 같은 Git remote(원격 연결 대상에 붙인 이름)를 지정할 수 있게 됐다. 이미 하나의 코드에 여러 원격 대상을 설정한 사람은 익숙한 이름으로 기본 대상을 고를 수 있다. 공지는 remote를 자동으로 만들거나 고쳐 준다고 말하지 않는다. 주소록의 ‘회사’나 ‘개인’이라는 기존 표지를 골라 전체 주소를 다시 쓰지 않는 것과 같다.

예상 밖 프로젝트 데이터로 도구가 갑자기 멈추지 않는다.

GitHub 프로젝트 데이터를 다룰 때 null project items(항목 자리는 있지만 읽을 값이 없는 상태)가 panic(처리하지 못한 오류로 프로그램이 갑자기 멈추는 일)을 일으키지 않게 됐다. 이는 기존 처리 경로의 수정이지 새 프로젝트 관리 기능은 아니다. 같은 버전에서 설치 안내의 Debian CLI 패키지 링크도 바로잡았다. 서류의 빈칸 하나를 만나도 직원이 그 칸을 건너뛰고 업무를 계속하며, 입구 안내판도 올바른 방향을 가리키는 것과 같다.

배포 과정과 기반 의존성을 유지보수했다.

평소 명령어를 바꿀 필요는 없지만, 이 버전은 OAuth, 서명, 빌드 검증과 여러 Go 의존성을 업데이트했다. CI(지속적 통합, 프로그램을 자동 검사하고 포장하는 과정)의 배포 작업도 정비해 GoReleaser를 올리고 Azure Code Signing을 갱신했으며, 각 빌드 작업에 태그를 붙이고 Windows에서 임시 태그를 만들 때 쓸 shell을 지정했다. 확장 기능 워크플로 템플릿의 actions/checkout도 업데이트됐다. 새 터미널 기능을 알린 것이 아니므로, 출고 공장을 계속 관리하는 변경으로 보는 것이 정확하다. 창고의 봉인기와 분류 규칙, 부품을 손보는 것과 같다.
