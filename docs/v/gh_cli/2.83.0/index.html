發行版本清單可辨認不可變版本

現在你用 gh release list 取得 JSON（方便其他程式讀取的固定格式資料）時，會多看到 isImmutable 欄位。它會標示這個發行版本是否不可變，也就是發出去後不該再被修改。若你用指令自動檢查版本或串接部署流程，就能先辨認哪些版本要當成定稿處理，少把仍可變動的版本和正式封存版混在一起。這像圖書館替每本書標上「可借閱」或「館藏定本」，一眼就知道哪本不能隨意改頁。

Original changelog
Add `isImmutable` to `release list` JSON output by @babakks in https://github.com/cli/cli/pull/12064

建立代理任務時可指定自訂代理

現在建立代理任務時，可以在 gh agent-task create 加上 --custom-agent 或 -a，指定要交給哪個自訂代理（替你執行特定工作的自動助手）。以前建立任務時不能在這一步選它；現在你可直接把適合的代理派去做對的工作，少一道建立後再調整的手續。這像寄件時就把信投進正確的專用信箱，不必先送到總收發室再轉寄。

Original changelog
`gh agent-task create`: support `--custom-agent`/`-a` flag by @BagToad in https://github.com/cli/cli/pull/12068

刪除倉庫前多一層提醒，也可還原 PR

現在你執行 gh repo delete --yes 卻沒有指定倉庫時，指令會提醒 --yes 被忽略，而不是讓你以為已略過確認。另新增 gh pr revert，可替一個 pull request（PR，把程式修改送來審查與合併的提案）建立反轉那次修改的操作。前者讓刪除前的判斷更清楚，後者讓已合併的改動有一條回頭路。這像拆房前先確認地址，裝錯零件後也能照說明把它換回來。

大型 Gist 檔案可正常查看與編輯

現在較大的 Gist（GitHub 上用來分享程式片段或文字檔的小型貼文）檔案，也能用指令查看與編輯。這是修復既有功能，不是新增另一種檔案服務；重點是原本因檔案較大而卡住的讀取與修改流程恢復可用。若你把長設定檔、記錄或程式碼放在 Gist，就不用為了大小另找工具處理。這像原本塞不進影印機的大張圖紙，現在能攤平後正常閱讀與修改。

Original changelog
fix(gist): add support for editing & viewing large files  by @luxass in https://github.com/cli/cli/pull/11761

驗證發行證明遇到 Sigstore 暫時不可用仍可運作

現在 gh attestation verify 驗證發行版本的 attestation（用來證明檔案來源與建立過程的簽署證明）時，即使 Sigstore 的 Public Good Instance 暫時無法使用，也能正常運作。指令同時改用 release 作為發行版本驗證的固定識別值。這是讓既有驗證流程在外部驗證服務短暫不可用時更可靠，不代表跳過驗證。這像查驗包裹封條時，主要櫃台暫時關閉，仍有正確的查驗方式可接手。

Release lists can identify immutable releases

When you use gh release list to get JSON, a structured format other programs can read, you now get an isImmutable field. It marks whether a release is immutable, meaning it should not be changed after publication. If you automate release checks or deployment workflows, you can distinguish a final archived release from one that may still change. It is like a library label showing which copy is the unalterable reference edition.

Original changelog
Add `isImmutable` to `release list` JSON output by @babakks in https://github.com/cli/cli/pull/12064

Create agent tasks with a chosen custom agent

When creating an agent task, you can now add --custom-agent or -a to gh agent-task create and choose a custom agent, an automated helper for a particular kind of work. You no longer need to create the task first and adjust its assignee afterward. It is like dropping a letter straight into the right specialist mailbox instead of routing it through a central mailroom.

Original changelog

Get a deletion warning and revert a pull request

If you run gh repo delete --yes without naming a repository, the command now warns that --yes is being ignored instead of making you think confirmation was skipped. A new gh pr revert command can also reverse the changes from a pull request, a proposal to review and merge code changes. One makes deletion intent clearer; the other gives merged work a route back. It is like checking the address before demolition and having instructions to undo a wrongly fitted part.

Large Gist files can be viewed and edited

You can now view and edit larger Gist files, the small GitHub posts used to share code snippets or text files, from the CLI. This repairs existing behavior rather than adding a new file service: reads and edits that failed because a file was large now work again. If you keep a long configuration file, log, or code sample in a Gist, you need not switch tools just because of its size. It is like finally being able to lay a large sheet flat in the copier and work on it.

Release-attestation checks keep working when Sigstore is unavailable

When gh attestation verify checks a release attestation, a signed record that proves a file's origin and build process, it can now keep working if Sigstore's Public Good Instance is temporarily unavailable. The command also uses release as the sentinel value for release-attestation verification. This makes the existing verification path more resilient when an outside service is unavailable; it does not mean skipping verification. It is like having a valid backup inspection desk when the main parcel-seal counter is briefly closed.
