可取得手動觸發工作流程的執行詳情

你手動啟動 GitHub Actions 工作流程（workflow，依步驟自動執行工作的設定）後，gh 現在能取得這次手動觸發執行的詳細資料。以前看執行紀錄時，手動按下啟動的那一趟可能少了可追查的資訊；現在工具會把它當成可辨識的一次執行來讀取。這讓你檢查部署、測試或其他自動化工作時，更容易確認剛剛啟動的是哪一次。就像叫了外送後能看到那一張訂單，而不是只知道店家曾經出餐。

REST API 固定使用指定版本

用 gh 與 GitHub 溝通時，REST API（讓程式透過網址交換資料的介面）現在固定使用 2022-11-28 這個版本。API 若隨服務端預設版本變動，原本可用的指令回應可能在沒有改動你的情況下改變；明確指定版本，工具就知道要依哪一套規則交談。這不是讓你多背一個日期，而是替日常指令減少不預期的差異。就像填表時固定拿同一版表格，不會今天欄位忽然換了位置。

建立 PR 時分叉只複製預設分支

Release builds now bundle license information

When you download a new gh release, its license information is now bundled during the release process. Licenses state how code may be used and distributed, so having them collected with the finished download makes it easier for a person or company to review what is included. This does not change any command you run; it simply makes each release package more complete. It is like getting the warranty and manual in the appliance box instead of requesting them from several manufacturers.

Filter project item lists with a query

When organizing a GitHub project board, gh project item-list can now take a --query flag, an option that tells a command how to act. You can use a query to select the items you need instead of retrieving the entire list and searching through it yourself. That helps on boards with many tasks when you only want one kind of item. The accompanying implementation was also cleaned up. It is like adding a search box to a library catalog before you walk to the shelf.

Retrieve details for manually dispatched workflow runs

After you manually start a GitHub Actions workflow, gh can now retrieve details for that dispatched run. A workflow is a configured sequence of automated steps. Previously, a run started with the manual button could have less traceable information when you reviewed executions. Now the tool recognizes and retrieves it as a specific run, making it easier to check which deployment, test, or automation job you just started. It is like being able to see your own delivery order instead of only knowing the restaurant sent food at some point.

Original changelog

Use a pinned REST API version

When gh communicates with GitHub, it now pins the REST API to version 2022-11-28. A REST API is an interface through which programs exchange data over web addresses. If the service's default API version changes, a command response can change even though you changed nothing. Naming one version tells gh which set of rules to use. You do not need to memorize the date; it simply reduces surprise differences in everyday commands, like always receiving the same version of a form.

Original changelog
Pin REST API version to 2022-11-28 by @williammartin in https://github.com/cli/cli/pull/12680

Workflow log views now preserve failure exit status

When checking a workflow run, gh run view now honors its --exit-status flag even when you ask for full logs or failed logs. The flag makes a command end with a success or failure result. Previously, combining it with --log or --log-failed could lose that signal, causing a script to misread whether the run failed. Now you can inspect the logs while automation still knows whether to stop or continue. It is like being able to read an alarm record while the warning light still stays on.

PR creation forks only the default branch

When gh pr create needs to fork a repository while creating a pull request, it now forks only the default branch instead of every branch. A pull request is a request to merge your changes into a project, and a fork is your own copy of someone else's repository. Since your change usually begins from the default branch, this narrows what must be copied for the task. Other branches are not treated as required baggage. It is like photocopying the main document you need rather than carrying home the whole filing cabinet.

Both areas also received performance and accessibility improvements, making the selection easier with assistive tools. It is like filling out a meeting invitation with a searchable attendee list and an AI assistant checkbox.

풀 리퀘스트, 즉 변경 사항을 프로젝트에 합쳐 달라는 신청을 편집할 때 대화형 안내에서 담당자를 고르고, 검색해 Copilot을 검토자로 추가할 수 있다. 대화형 안내는 명령어가 선택지를 단계별로 묻는 화면이므로 계정 이름을 모두 외워 입력할 필요가 없다. 두 기능은 성능과 접근성도 개선했다. 회의 초대장에서 참석자 명단을 검색하고 AI 도우미에 체크하는 것과 같다.

릴리스 업로드와 gist 편집 안내를 더 명확하게 제공

릴리스 자산을 올릴 때 gh release upload는 --clobber 플래그가 기존 자산을 먼저 삭제한 뒤 다시 올린다는 점을 더 분명히 설명한다. gh gist edit에는 짧은 코드나 글을 공유하는 GitHub 게시물인 gist를 편집하는 사용 예도 추가했다. 영어 오타 하나도 고쳤다. 명령어의 핵심 기능은 바꾸지 않지만, 실수할 수 있는 행동을 하기 전에 뜻을 알기 쉽게 한다. 새 파일을 넣기 전에 낡은 수납장을 비운다는 버튼 안내문과 같다.

내부 유지보수 업데이트로 이후 개발과 릴리스를 안정화

이번 릴리스는 gh의 내부 유지보수도 정리했다. 피드백 issue 틀을 없애고 issue와 PR 분류를 공유 워크플로로 옮겼으며, 기능 감지 주석을 보태고 Go 올리기 워크플로를 수동으로 시작할 수 있게 했다. 오류 형식과 문서 검사도 고쳤고 Go, OAuth, 텍스트, 터미널, 서명, 릴리스 관련 의존성을 업데이트했으며 새 기여자도 기록했다. 사용자가 입력할 새 명령어가 아니라 개발·테스트·릴리스의 연결을 유지하는 변경이다. 새 메뉴를 늘리지 않고 주방 배관과 준비 과정을 정돈하는 것과 같다.
