建立與查看協作項目更好找人、更容易辨識

你現在建立 pull request（請別人檢查並合併程式修改的申請）時，可依登入帳號請求 reviewer（審查修改的人）協助，也能在互動選單搜尋後挑選。查看 pull request 或 issue（工作或問題追蹤單）時，所有參與者會顯示較好懂的名稱；關閉重複 issue 時還能用 --duplicate-of 指向原始單。這像替辦公室的待辦貼上人名與原件連結，接手的人不必猜誰是誰、哪張才是正本。

Original changelog
`gh pr create`: login-based reviewer requests and search-based interactive selection by @BagToad in https://github.com/cli/cli/pull/12627
`gh pr view` and `gh issue view`: show friendly display names for all actors by @BagToad in https://github.com/cli/cli/pull/12854
`gh issue close`: add `--duplicate-of` flag and duplicate reason by @tksohishi in https://github.com/cli/cli/pull/12811

篩選差異、讀取資料與啟動 Copilot 更順手

你查看 pull request 的 diff（兩份修改內容的逐行差異）時，現在可用 --exclude 略過不想看的檔案；view 或 list 的 JSON（讓其他工具讀取的固定格式資料）檔案資料也多了 changeType，可辨認變更類型。列出或查看 agent-task（交給代理程式的任務）同樣能輸出 JSON，而啟動 gh copilot 時會設定 COPILOT_GH 環境變數，讓 Copilot CLI 知道自己由 gh 叫起。像整理一疊校稿：先抽走不必看的頁面，再用標籤交給下一位助手。

`gh pr view/list`: add `changeType` field to files JSON output by @yuvrajangadsingh in https://github.com/cli/cli/pull/12657

複製、設定合併訊息與追查程式來源多了控制選項

你用 gh repo clone（把遠端專案複製到電腦）取得專案時，可用 --no-upstream 不設定上游來源；編輯 repository（程式專案存放處）時，能指定 squash merge（把多次修改壓成一筆再合併）的提交訊息。gh browse 也新增 --blame，可直接開啟逐行追查紀錄，看看某一行是誰何時改的。這些選項像搬家時能選擇不留下原屋轉寄地址、替整箱物品寫一張總標籤，還能查出哪位室友放進那件東西。

編輯與輸出資料時少遇到錯誤與亂碼

你編輯只有標題或內容的 Draft Issue（尚未正式送出的草稿問題單）時，gh project item-edit 不再出錯；安裝 extension（外掛工具）失敗時，也會顯示 owner/repo 而不是難懂的內部資料。--json 輸出的 assignee（被指派處理的人）databaseId 不會再一直是 0，顏色化 JSON 與 diff 的 ANSI SGR 控制碼也已修正。另修好 port forwarder（把本機連到遠端連接埠的轉送工具）整數轉換、帶 repository 參數使用 --remote 的錯誤，以及 issue comments 的多餘 API 呼叫。像修好一張表單：欄位、標籤和傳遞管線終於各自回到正確位置。

Create and view collaboration items with clearer people and choices

You can now create a pull request and ask reviewers based on their login, or search for them in the interactive picker. Pull request and issue views show friendlier display names for every participant, and closing a duplicate issue can point back to the original with --duplicate-of. It is like labelling office tasks with names and a link to the original sheet, so the next person does not have to guess who is who or which copy matters.

Filter diffs, read data, and launch Copilot more smoothly

You can now hide files you do not want to inspect from a pull request diff with --exclude. JSON output for pull request files adds changeType, and listing or viewing agent tasks can also return JSON for other tools to read. When gh launches Copilot CLI, it sets the COPILOT_GH environment variable so Copilot knows who started it. It is like sorting a stack of proofs: remove the pages you do not need, then add labels for the next helper.

Get more control over cloning, merge messages, and code history

When you use gh repo clone to copy a remote project to your computer, --no-upstream lets you avoid setting an upstream. Editing a repository can now set the commit message for a squash merge, which combines several changes into one. gh browse also adds --blame to open a line-by-line history showing who changed a line and when. It is like moving house with the option not to leave a forwarding address, labelling one whole box, and checking which roommate packed an item.

Hit fewer errors and garbled results when editing or exporting data

You should hit fewer rough edges when editing and exporting data. gh project item-edit no longer errors for a Draft Issue with only a title or body; a failed extension install now shows owner/repo instead of raw internal data; and assignee databaseId values in --json output no longer stay at zero. Color codes in JSON and diffs, port-forwarding number conversion, --remote with a repository argument, and an unnecessary API call for issue comments were also fixed. It is like repairing a form so its fields, labels, and delivery tubes all return to the right place.
