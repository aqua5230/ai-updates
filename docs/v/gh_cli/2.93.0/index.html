身分驗證與 API 檢查更準了

這批修的是幾個跟認證、API 檢查有關的準確度問題。以前 secrets（機密資訊，例如 API 金鑰）只能綁在使用者或 repository（儲存庫）上，現在也能讓 agent（代理，例如自動化機器人）以 application（應用程式身分）的名義使用 secrets。查詢 PR（pull request，程式碼合併請求）時，以前有個「只認編號」的優化捷徑會跳過 API 驗證，可能讓查詢結果不準，現在拿掉這個捷徑，改成每次都好好驗證。以前遇到 401（未授權）錯誤只丟出一個看不懂的錯誤，現在會直接印出「請跑 gh auth refresh」提醒你該重新驗證登入了。gh copilot 指令執行失敗時，以前只丟錯誤訊息，現在會提示你直接跑 copilot 指令排查。像 ATM 提款失敗時不再只顯示錯誤代碼，而是直接告訴你下一步該怎麼做。

Authentication and API checks get more accurate

This batch fixes a handful of accuracy issues around authentication and API checks. Secrets (sensitive values like API keys) used to be scoped only to a user or a repository; now an agent (an automated bot, for example) can also use secrets under an application identity. Looking up a PR (pull request) by number used to take a "number-only" shortcut that skipped API validation, which could return inaccurate results; that shortcut is gone, so every lookup is validated now. Hitting a 401 (unauthorized) error used to just print an unhelpful error; now it prints "run gh auth refresh" to tell you exactly what to do. When `gh copilot` fails to run, it now hints you to run the copilot command directly to troubleshoot, instead of just showing an error. It's like an ATM that no longer just flashes an error code when a withdrawal fails, but tells you directly what to do next.

Supply-chain security tightens up; releases get locked

This batch strengthens supply-chain security (making sure the software you download hasn't been tampered with). `gh release verify` (the command that checks a release's integrity) now derives the right digest algorithm (the hashing method used to verify content hasn't changed) from the length of the ref (version tag), making verification more accurate. The CLI's own build pipeline switched its verification action from `actions/attest-build-provenance` to `actions/attest` (another official GitHub-hosted attestation action). In the project's own GitHub Actions (automation scripts), first-party actions (ones GitHub itself maintains) are now pinned by SHA (a fixed version hash), so they can't silently change content if the upstream action gets updated. Docs also now note that starting with v2.93.0, GitHub Releases become immutable (locked after publishing — no more edits). It's like sealing a shipment before it leaves the factory and using an inspector who only checks a fixed batch number, so the goods can't be swapped mid-transit.

This batch is installation-docs and small polish fixes. The official install command docs were updated; some Linux distros (the dnf family) had an unnecessary `--repo gh-cli` in their install commands, which is now removed; on Windows, running certain commands used to flash a tzutil console window in the background — that flicker is fixed; and a duplicated word plus a grammar typo in the docs were also corrected. The rest is extended PR-screening process work, telemetry (usage-reporting statistics) adjustments, and dependency version bumps that don't affect day-to-day use. It's like proofreading a manual to remove extra filler words and typos in one pass — everything just reads cleaner now.

認証と API チェックの精度が上がった

今回は認証と API チェックまわりの精度を直したバッチだ。以前 secrets（機密情報、たとえば API キー）はユーザーか repository（リポジトリ）にしか紐付けられなかったが、今は agent（代理、たとえば自動化ボット）が application（アプリケーション）としての身分で secrets を使えるようになった。PR（pull request）を番号で調べる際、以前は「番号だけを信用する」最適化のせいで API 検証を飛ばし、結果が不正確になることがあったが、その近道は取り除かれ、毎回きちんと検証されるようになった。401（未認可）エラーに遭遇すると、以前はわかりにくいエラーだけが表示されていたが、今は「gh auth refresh を実行してください」とそのまま印字される。gh copilot コマンドの実行に失敗すると、以前はエラーだけだったが、今は copilot コマンドを直接実行して確認するよう案内するようになった。ATM が引き出し失敗時にエラーコードだけを出す代わりに、次に何をすべきかを直接教えてくれるようなものだ。

サプライチェーンセキュリティを強化、リリースはロック不可変に

今回はサプライチェーンセキュリティ（ダウンロードしたソフトウェアが改ざんされていないことを保証する仕組み）の強化。gh release verify（リリースの完全性を検証するコマンド）は、ref（バージョンタグ）の長さから使うべき digest algorithm（ハッシュアルゴリズム、内容が改ざんされていないか検証する方法）を自動判定するようになり、検証がより正確になった。CLI 自体のビルドパイプラインは、検証に使う actions/attest-build-provenance を actions/attest（GitHub がホストする別の認証アクション）に切り替えた。プロジェクト自身の GitHub Actions（自動化スクリプト）のうち、first-party（GitHub 自身が保守する）のアクションは今、SHA（固定バージョンのハッシュ値）でバージョンを固定するようになり、アップストリームが更新されても内容が勝手に変わらなくなった。ドキュメントには v2.93.0 から GitHub Releases が immutable（不変、公開後は内容がロックされ変更不可）になることも追記された。工場出荷前に封印をもう一重かけ、決まったロット番号しか通さない検査官を置いたようなもので、途中で中身をすり替えられなくなる。

인증과 API 검사가 더 정확해졌다

이번엔 인증과 API 검사 쪽의 정확도 문제 몇 가지를 고쳤다. 예전엔 secrets(민감 정보, 예를 들어 API 키)를 사용자나 repository(저장소)에만 묶을 수 있었는데, 이제 agent(에이전트, 예를 들어 자동화 봇)도 application(애플리케이션) 신분으로 secrets를 쓸 수 있다. PR(pull request)을 번호로 조회할 때 예전엔 '번호만 믿는' 최적화 지름길 때문에 API 검증을 건너뛰어 결과가 부정확할 수 있었는데, 이 지름길을 없애 매번 제대로 검증한다. 401(권한 없음) 오류를 만나면 예전엔 알아보기 힘든 오류만 떴는데, 이제 'gh auth refresh를 실행하세요'라고 바로 알려준다. gh copilot 명령 실행이 실패하면 예전엔 오류만 던졌는데, 이제 copilot 명령을 직접 실행해 확인해 보라고 안내한다. ATM이 출금 실패 때 오류 코드만 보여주는 대신, 다음에 무엇을 해야 하는지 바로 알려주는 것과 같다.

공급망 보안 강화, 릴리스는 잠겨서 못 바꾼다

이번엔 공급망 보안(다운로드한 소프트웨어가 변조되지 않았음을 보장하는 장치)을 강화했다. gh release verify(릴리스 무결성을 검사하는 명령)는 이제 ref(버전 태그) 길이로 어떤 digest algorithm(해시 알고리즘, 내용이 변조되지 않았는지 검증하는 방법)을 쓸지 자동 판단해 검증이 더 정확해졌다. CLI 자체 빌드 파이프라인은 검증에 쓰던 actions/attest-build-provenance를 actions/attest(GitHub가 호스팅하는 또 다른 인증 액션)로 바꿨다. 프로젝트 자체 GitHub Actions(자동화 스크립트) 중 first-party(GitHub가 직접 관리하는) 액션은 이제 SHA(고정 버전 해시값)로 버전을 고정해서, 업스트림이 바뀌어도 내용이 몰래 달라지지 않는다. 문서에도 v2.93.0부터 GitHub Releases가 immutable(불변, 공개 후 잠겨서 더는 수정 불가)이 된다고 명시됐다. 출고 전 포장에 봉인을 한 겹 더 하고, 정해진 로트 번호만 통과시키는 검수원을 두어, 중간에 내용물이 바뀌지 않게 한 셈이다.
