Windows 系統匣不再鬧雙胞或留白，面板卡片排序全同步

現在 Windows 的面板與系統匣（system tray，常駐在螢幕右下角的圖示選單）畫面變得更順眼了。當你拖曳額度卡（quota card）改變排列順序並重載主題時，系統會先重新讀取共用的已儲存排序（shared saved order），所以不管你打開哪個支援拖曳排序的面板，上面的 Claude、Codex 與 Antigravity 卡片順序都會是一致的。另外，以前系統匣在啟動時，會過早對還沒準備好的視窗定位，硬是把一個空白的白色或深色長方形拉到畫面上；如果重複啟動兩次，第二個實例還會與第一個爭奪共用的 WebView2（讓桌面程式內嵌顯示網頁畫面的元件）用來存放使用者資料的目錄而初始化失敗，留下一塊純白長方形殘留在螢幕上。現在系統改為顯示前一刻才定位視窗，而且面板顯示後如果又切換面板，視窗定位還是會照舊重新套用一次；系統匣會持有一個具名 mutex（named mutex，用來確保同時間只有一個程式運行的鎖定機制），點第二次只會顯示「已在執行中」的在地化提示然後關閉。就像一間店面，以前店員還沒把假人模特兒穿好衣服就把鐵捲門拉開，或者兩位店長在同一個地址搶著掛招牌；現在他們會在開門前把商品陳列整齊，而且知道這地址有人營業就會客氣地離開，不再鬧雙胞。

**Windows panel card order now follows across themes**: reloading a theme after dragging a quota card now rereads the shared saved order before state injection, so every draggable panel opens with the same Claude, Codex, and Antigravity card order.

狀態列額度同步更安全，Windows 不掉額度資料、macOS 不閃退

當你同時開好幾個 Claude Code 視窗時，你的額度資料（`rate_limits`）不再莫名其妙遺失，這波修復也順手解決了 macOS 使用者一開程式就閃退的災難。以前在 Windows 上，如果有兩個實例同時刷新狀態列，用來保護檔案不被同時修改的檔案鎖（file lock）是採用非阻塞式（non-blocking，搶不到鎖就直接放棄並拋出錯誤），而這個搶鎖失敗拋出的 OSError 錯誤又被系統吞掉，導致讀取並寫入 `rate_limits`（額度資料）的過程完全沒有同步保護，把資料給弄丟了。現在 Windows 會不斷輪詢（polling，定期重複嘗試）直到取得鎖，最多等 10 秒；若該檔案系統真的不支援鎖定，就會退回單靠原子寫入（atomic write，確保寫入動作一次完成中途不被打斷）。實測 200 組並行（concurrent）運作，資料遺失次數從 21 次降為完美的 0 次。而這次在修復 Windows 鎖定問題時，不小心用了一個 macOS 上不存在的 `errno.EDEADLOCK` 常數，導致所有 macOS 使用者一載入程式就直接崩潰；現在已經補上防護，當找不到該屬性時會自動退回使用 `errno.EDEADLK`。就像多個記帳員同時要寫同一本帳冊，以前在 Windows 上大家不排隊硬搶，搶不到筆就漏記而弄丟了原本的紀錄，而在 macOS 上則是記帳員看到新版的排隊規定看不懂就直接罷工；現在大家不僅學會耐心排隊等筆，看不懂新規定時也懂得變通，用自己熟悉的方式繼續工作。

Original changelog
**The status line no longer drops quota data when two Claude Code instances refresh at once on Windows**: the hook's file lock was blocking on POSIX (`fcntl.flock(LOCK_EX)`) but non-blocking on Windows (`msvcrt.locking(LK_NBLCK)`), and a contended lock raised `OSError` that was swallowed — leaving the `rate_limits` carry-forward read-modify-write in `save()` completely unsynchronized. Windows now polls the lock until it is acquired (10s deadline), matching POSIX semantics; genuinely unsupported locking still falls back to the atomic write alone. Measured over 200 concurrent hook pairs: `rate_limits` were lost 21 times before, 0 times after.
**The Claude Code hook no longer crashes on macOS**: a constant introduced by the Windows lock fix above referenced `errno.EDEADLOCK`, which doesn't exist on macOS — importing `usage_statusline.py` failed immediately for every macOS user. Now falls back to `errno.EDEADLK` when `EDEADLOCK` isn't defined.

Windows system tray no longer blanks out or duplicates, and panel cards sync flawlessly

The Windows panels and system tray (the icon menu tucked in the corner of your taskbar) now look much cleaner. When you drag a quota card to change its order and then reload a theme, the system now rereads the shared saved order first, so your Claude, Codex, and Antigravity cards will line up exactly the same way across all draggable panels. Previously, the tray would prematurely position its hidden window before it finished rendering, dragging an empty white or dark rectangle onto your screen at launch; furthermore, if you accidentally launched the tray twice, the second instance would fight the first over the shared WebView2 (the component that renders web content within desktop apps) user-data directory, fail to initialize, and linger as a bare white rectangle. It's like a storefront where employees used to roll up the shutters before dressing the mannequins, or two managers trying to open a shop at the same address; now they perfectly arrange the display before opening the doors, and the second manager politely leaves upon seeing the "Open" sign.

Safer quota sync prevents quota data drops on Windows and startup crashes on macOS

Your quota data will no longer vanish when you run multiple Claude Code windows at the same time, and a related fix resolves a disastrous startup crash for macOS users. Previously on Windows, if two instances refreshed the status line simultaneously, the file lock (a mechanism preventing simultaneous file edits) operated in a non-blocking manner (immediately giving up if the lock wasn't available). The resulting OSError was silently swallowed, leaving the read-modify-write cycle for `rate_limits` (quota data) completely unsynchronized, causing data drops. Windows now polls (repeatedly checks) for the lock until acquired, with a 10-second deadline; if the file system genuinely doesn't support locking, it safely falls back to an atomic write (saving the entire change in one uninterruptible motion). In a test of 200 concurrent runs, data loss plummeted from 21 incidents to zero. Furthermore, the initial fix for this Windows lock issue introduced an `errno.EDEADLOCK` constant that simply doesn't exist on macOS, causing every macOS user's app to crash instantly upon loading. It now falls back to the supported `errno.EDEADLK` when the former is undefined. It's like multiple accountants trying to write in the same ledger: previously on Windows they'd just grab at the book, fail to get a pen, and lose track of the records, while on macOS the accountants saw the new queuing rules, got confused, and walked off the job entirely; now they patiently wait in line for the pen, and know how to adapt when the rulebook uses unfamiliar terms.

利用枠の同期が安全になり、Windows での利用枠データの消失と macOS でのクラッシュを防止

複数の Claude Code のウィンドウを同時に開いた際、利用枠のデータが突然消滅することがなくなり、それに伴う macOS ユーザーの起動時クラッシュの惨事も解決された。以前、Windows では 2 つのインスタンスが同時にステータス行を更新しようとすると、ファイルの同時編集を防ぐファイルロック（file lock）が非ブロッキング（non-blocking、ロックが取れないとすぐに諦めてエラーを投げる方式）に動作し、しかもその際に生じた OSError がシステムに握りつぶされていた。その結果、`rate_limits`（利用枠データ）の読み取り・変更・書き込みがまったく同期されず、データを消失させていた。現在は Windows でもロックが取れるまで最大 10 秒間ポーリング（polling、定期的な再試行）を行うようになり、ロック自体をサポートしていないファイルシステムの場合は、原子書き込み（atomic write、書き込み処理を途中で分断させず一気に完了させる仕組み）のみにフォールバックする。200 組の並行（concurrent）動作テストでは、データ消失回数が 21 回から 0 回へと見事に改善された。また、この Windows のロック修正を行った際、macOS には存在しない `errno.EDEADLOCK` という定数が混入し、macOS ユーザー全員がプログラムを読み込んだ瞬間にクラッシュするようになっていたが、現在は定義が見つからない場合に `errno.EDEADLK` へフォールバックするよう修正された。複数の会計士が同じ帳簿に記入する際、以前の Windows では順番待ちをせずに奪い合い、ペンを取れずに記録を失わせてしまい、macOS では新しい順番待ちのルールが読めずに全員ボイコットしていたのが、今は辛抱強く列に並び、見慣れないルールが来ても機転を利かせて仕事を続けられるようになったようなものである。

Windows 시스템 트레이의 빈 화면이나 중복 실행이 사라지고 패널 카드 순서가 완벽히 동기화됨

Windows의 패널과 시스템 트레이(system tray, 화면 오른쪽 아래에 떠 있는 아이콘 메뉴) 화면이 훨씬 깔끔해졌다. 쿼터 카드(quota card, 할당량 카드)를 드래그해서 순서를 바꾼 뒤 테마를 다시 불러오면, 시스템이 공유된 저장 순서(shared saved order)를 먼저 다시 읽어오기 때문에 드래그 가능한 모든 패널에서 Claude, Codex, Antigravity 카드가 똑같은 순서로 나타난다. 또한, 예전엔 시스템 트레이가 켜질 때 아직 준비되지 않은 숨겨진 창의 위치를 너무 일찍 잡는 바람에 빈 하얀색이나 어두운 직사각형이 화면에 튀어나오곤 했다. 게다가 두 번 중복해서 실행하면 두 번째 프로그램이 첫 번째와 공유되는 WebView2(데스크톱 앱 내에 웹 화면을 표시하는 구성 요소) 사용자 데이터 폴더를 두고 싸우다가 초기화에 실패해 새하얀 직사각형 화면만 덩그러니 남기도 했다. 이제는 창을 화면에 보여주기 직전에 위치를 잡도록 바뀌었고(패널이 표시된 후 다른 패널로 전환할 때도 위치가 이전처럼 다시 적용됨), 시스템 트레이가 이름 있는 뮤텍스(named mutex, 한 번에 하나의 프로그램만 실행되도록 보장하는 잠금 장치)를 쥐고 있어서 두 번 누르면 현지화된 '이미 실행 중'이라는 알림만 띄우고 조용히 종료된다. 가게 직원이 마네킹에 옷도 입히기 전에 셔터부터 올려 텅 빈 진열장을 보여주거나, 두 점장이 같은 주소에서 간판을 달겠다고 싸우던 것이, 이제는 문을 열기 전에 상품을 가지런히 정리하고 다른 사람이 이미 영업 중인 걸 보면 조용히 발길을 돌리는 것과 같다.

Original changelog
**Windows panel card order now follows across themes**: reloading a theme after dragging a quota card now rereads the shared saved order before state injection, so every draggable panel opens with the same Claude, Codex, and Antigravity card order.
**The tray panel no longer appears as an empty white/dark rectangle at launch**: pywebview's `resize()` and `move()` call `SetWindowPos` with `SWP_SHOWWINDOW`, and the tray placed its still-hidden window as soon as the panel document loaded — dragging the unrendered panel onto the screen on every start. The window is now placed right before it is shown; after a visible panel switch the placement is re-applied as before.
**Launching the Windows tray twice no longer leaves a blank white window**: a second instance fought the first over the shared WebView2 user-data directory, so its panel failed to initialize and lingered on screen as a bare white rectangle. The tray now holds a named mutex for its lifetime; a second launch shows an "already running" notice (localized) and exits instead.

할당량 동기화가 안전해져 Windows의 할당량 데이터 유실과 macOS의 강제 종료 방지

이제 여러 개의 Claude Code 창을 동시에 띄워도 할당량 데이터가 허무하게 날아가지 않으며, 이 과정에서 macOS 사용자가 프로그램을 켜자마자 튕기던 재앙도 함께 해결되었다. 예전 Windows 환경에서는 두 개의 프로그램이 동시에 상태 표시줄을 새로 고치려 할 때, 파일이 동시에 수정되는 것을 막는 파일 잠금(file lock) 장치가 비차단식(non-blocking, 잠금을 얻지 못하면 곧바로 포기하고 오류를 던지는 방식)으로 작동했고, 이때 발생하는 OSError마저 시스템이 꿀꺽 삼켜 버렸다. 그 결과 `rate_limits`(할당량 데이터)를 읽고 수정하고 쓰는 과정이 전혀 동기화되지 않아 데이터가 유실되기 일쑤였다. 이제 Windows는 잠금을 얻을 때까지 최대 10초 동안 계속 폴링(polling, 주기적으로 다시 시도함)하며, 파일 시스템이 잠금을 아예 지원하지 않으면 원자적 쓰기(atomic write, 중간에 끊기지 않고 한 번에 쓰기를 완료하는 방식)로 돌아가 안전을 확보한다. 200개를 병행(concurrent)으로 돌려본 결과, 데이터 유실이 21번에서 0번으로 완벽하게 사라졌다. 한편, 이 Windows 잠금 문제를 고치면서 macOS에는 있지도 않은 `errno.EDEADLOCK` 상수를 무심코 쓰는 바람에 모든 macOS 사용자가 프로그램을 켜자마자 죽어버리는 사태가 있었는데, 이제는 해당 속성이 없으면 `errno.EDEADLK`를 대신 쓰도록 방어막을 쳤다. 여러 명의 회계사가 같은 장부에 기록을 남기려고 할 때, 예전 Windows에서는 줄을 서지 않고 장부를 빼앗다 펜을 못 잡아 기록을 빼먹어 버렸고 macOS에서는 새로 바뀐 줄 서기 규칙을 이해하지 못해 아예 파업해 버렸던 반면, 이제는 펜이 날 때까지 끈기 있게 기다리고 규칙이 낯설어도 융통성 있게 자기 할 일을 계속하는 것과 같다.

Original changelog
**The status line no longer drops quota data when two Claude Code instances refresh at once on Windows**: the hook's file lock was blocking on POSIX (`fcntl.flock(LOCK_EX)`) but non-blocking on Windows (`msvcrt.locking(LK_NBLCK)`), and a contended lock raised `OSError` that was swallowed — leaving the `rate_limits` carry-forward read-modify-write in `save()` completely unsynchronized. Windows now polls the lock until it is acquired (10s deadline), matching POSIX semantics; genuinely unsupported locking still falls back to the atomic write alone. Measured over 200 concurrent hook pairs: `rate_limits` were lost 21 times before, 0 times after.
**The Claude Code hook no longer crashes on macOS**: a constant introduced by the Windows lock fix above referenced `errno.EDEADLOCK`, which doesn't exist on macOS — importing `usage_statusline.py` failed immediately for every macOS user. Now falls back to `errno.EDEADLK` when `EDEADLOCK` isn't defined.
